# Observability — Monitoring, Logs, Errors, Alerts

Every project generated by this factory gets observability from day one. Not optional. Not post-launch.

The three pillars:
- **Logs** — what happened (structured, searchable, retained)
- **Metrics** — how the system is behaving (latency, error rate, throughput)
- **Traces** — why something is slow (distributed request tracing)
- **Errors** — what broke and where (crash reports, exceptions, stack traces)
- **Alerts** — who gets woken up and when

---

## The Golden Rule: Instrument Once, Backend Flexible

Use **OpenTelemetry** (OTel) as the instrumentation standard across all layers. OTel is vendor-neutral — you write the same instrumentation code and point it at:
- CloudWatch (AWS-native, free with ECS)
- Grafana Cloud (open source, generous free tier)
- Datadog (powerful, expensive)
- Honeycomb / SigNoz (OTel-native)

This means you can start with CloudWatch and migrate to Datadog later without touching application code.

## Tier Selection by Scale & Budget

### Tier 1: MVP (zero extra cost, AWS-native)
Best for: new projects, <$50/month ops budget, single-person teams

| Layer | Tool | Cost |
|---|---|---|
| API logs | AWS CloudWatch Logs (already configured in ECS task) | Included in ECS |
| API metrics | CloudWatch Metrics (ECS built-in: CPU, memory, request count) | Included |
| API errors | Sentry free tier (5k errors/month) | Free |
| Infra | CloudWatch alarms + AWS CloudTrail | ~$5/month |
| Mobile crashes | Firebase Crashlytics | Free |
| Mobile performance | Firebase Performance Monitoring | Free |
| Alerts | CloudWatch Alarms → SNS → email/SMS | ~$1/month |
| Uptime | AWS Route 53 health checks OR UptimeRobot free | Free |

Setup effort: 2–3 hours. No new accounts needed beyond Firebase.

→ See [TIER1-MVP.md](TIER1-MVP.md) for step-by-step setup.

---

### Tier 2: Growth (best price/value, ~$30–80/month)
Best for: paying customers, team of 2–5, need dashboards + traces

| Layer | Tool | Cost |
|---|---|---|
| All logs + metrics + traces | Grafana Cloud (free tier: 50GB logs, 10k metrics, 50GB traces/month) | Free–$29/month |
| API errors | Sentry Team ($26/month, 100k errors) | $26/month |
| Infra | Grafana + Prometheus (on ECS as sidecar or Fargate task) | Compute cost only |
| Mobile crashes | Sentry (same account, Flutter SDK) | Included above |
| Alerts | Grafana Alerting (built-in) | Free |
| Uptime | Grafana Synthetic Monitoring | Free tier |

This is the **recommended default** for projects graduating from MVP. The Grafana LGTM stack (Loki + Grafana + Tempo + Mimir) is production-grade and free at this scale.

→ See [TIER2-GROWTH.md](TIER2-GROWTH.md) for step-by-step setup.

---

### Tier 3: Scale / Enterprise ($200+/month)
Best for: >10k DAU, on-call rotations, compliance requirements, multi-region

| Layer | Tool | Cost |
|---|---|---|
| Full platform | Datadog (~$15/host/month infra + $31/host APM) | $46+/host/month |
| Alternative | Grafana Cloud Pro (~$2500/month at scale) | ~$2500/month |
| Audit logs | AWS CloudTrail + CloudTrail Lake | $2–5/GB |
| Security events | AWS Security Hub + GuardDuty | $5–30/month |
| Mobile | Datadog Mobile (or keep Sentry) | Varies |

Realistic: most projects never need Tier 3 until $1M+ ARR.

## Layer-by-Layer Setup

### API / Backend (Node.js ECS)

#### Structured Logging (all tiers)
Use `pino` — fastest Node.js JSON logger. Never use `console.log` in production.

```javascript
// backend/src/config/logger.js
import pino from 'pino'

export const logger = pino({
  level: process.env.LOG_LEVEL || 'info',
  base: { service: process.env.SERVICE_NAME || 'api' },
  timestamp: pino.stdTimeFunctions.isoTime,
  // In production, ECS ships logs to CloudWatch automatically
  // JSON format is required for CloudWatch Logs Insights queries
})

// Usage in controllers:
// logger.info({ userId: req.user.id, habitId: id }, 'habit.created')
// logger.error({ err, userId: req.user.id }, 'habit.create.failed')
```

Add to package.json: `"pino": "^9.x"`

ECS task definition already has CloudWatch log driver — JSON logs from pino go directly to CloudWatch Logs.

#### Error Tracking — Sentry (all tiers)
```javascript
// backend/src/config/sentry.js
import * as Sentry from '@sentry/node'
import { nodeProfilingIntegration } from '@sentry/profiling-node'

Sentry.init({
  dsn: process.env.SENTRY_DSN,
  environment: process.env.NODE_ENV,
  integrations: [nodeProfilingIntegration()],
  tracesSampleRate: process.env.NODE_ENV === 'production' ? 0.1 : 1.0,
  profilesSampleRate: 0.1,
})

export { Sentry }
```

```javascript
// In index.js — BEFORE routes
import './config/sentry.js'
// AFTER routes — Sentry error handler must be last
app.use(Sentry.expressErrorHandler())
```

Environment variable needed: `SENTRY_DSN` — add to AWS Secrets Manager + ECS task definition.

#### OpenTelemetry Tracing (Tier 2+)
```javascript
// backend/src/config/telemetry.js (loaded before everything else)
import { NodeSDK } from '@opentelemetry/sdk-node'
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http'
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node'

const sdk = new NodeSDK({
  traceExporter: new OTLPTraceExporter({
    url: process.env.OTEL_EXPORTER_OTLP_ENDPOINT, // Grafana or Datadog endpoint
  }),
  instrumentations: [getNodeAutoInstrumentations()],
  serviceName: process.env.SERVICE_NAME || 'api',
})

sdk.start()
```

`getNodeAutoInstrumentations()` auto-instruments: HTTP, Express, pg, Redis, DNS — zero manual code.

### Infrastructure (AWS)

#### CloudWatch Alarms (all tiers, add to infra/modules/ecs/main.tf)
```hcl
resource "aws_cloudwatch_metric_alarm" "api_error_rate" {
  alarm_name          = "${var.project}-api-5xx-rate"
  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = 2
  metric_name         = "HTTPCode_Target_5XX_Count"
  namespace           = "AWS/ApplicationELB"
  period              = 60
  statistic           = "Sum"
  threshold           = 10
  alarm_description   = "API 5xx error rate exceeding threshold"
  alarm_actions       = [aws_sns_topic.alerts.arn]
  dimensions = {
    LoadBalancer = var.alb_arn_suffix
    TargetGroup  = var.target_group_arn_suffix
  }
}

resource "aws_cloudwatch_metric_alarm" "ecs_cpu_high" {
  alarm_name          = "${var.project}-ecs-cpu-high"
  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = 3
  metric_name         = "CPUUtilization"
  namespace           = "AWS/ECS"
  period              = 60
  statistic           = "Average"
  threshold           = 80
  alarm_actions       = [aws_sns_topic.alerts.arn]
  dimensions = {
    ClusterName = "${var.project}-cluster"
    ServiceName = "${var.project}-api"
  }
}

resource "aws_sns_topic" "alerts" {
  name = "${var.project}-alerts"
}

resource "aws_sns_topic_subscription" "email" {
  topic_arn = aws_sns_topic.alerts.arn
  protocol  = "email"
  endpoint  = var.alert_email
}
```

#### AWS CloudTrail (Tier 1+, add to infra/main.tf)
```hcl
resource "aws_cloudtrail" "main" {
  name                          = "${var.project}-trail"
  s3_bucket_name                = aws_s3_bucket.cloudtrail.id
  include_global_service_events = true
  is_multi_region_trail         = false
  enable_log_file_validation    = true

  event_selector {
    read_write_type           = "WriteOnly"  # Only log write operations (cost control)
    include_management_events = true
  }
}

resource "aws_s3_bucket" "cloudtrail" {
  bucket = "${var.project}-cloudtrail-logs"
}

resource "aws_s3_bucket_lifecycle_configuration" "cloudtrail" {
  bucket = aws_s3_bucket.cloudtrail.id
  rule {
    id     = "expire-old-logs"
    status = "Enabled"
    filter {}
    expiration { days = 90 }
  }
}
```

CloudTrail records every AWS API call: who created/deleted/modified what, from which IP, at what time. Essential for security audits and incident response. Costs ~$2/100k events.

#### RDS Performance Insights (add to infra/modules/rds/main.tf)
```hcl
resource "aws_db_instance" "postgres" {
  # ... existing config ...
  performance_insights_enabled          = true
  performance_insights_retention_period = 7  # days, free tier
  monitoring_interval                   = 60  # Enhanced monitoring every 60s
  monitoring_role_arn                   = aws_iam_role.rds_monitoring.arn
}
```

This adds real-time query performance monitoring to RDS — shows slow queries, wait events, connection count. Free for 7-day retention.

### Mobile (Flutter)

#### Firebase Crashlytics (all tiers — free)
```yaml
# pubspec.yaml additions
dependencies:
  firebase_core: ^3.x
  firebase_crashlytics: ^4.x
  firebase_performance: ^0.10.x
```

```dart
// lib/main.dart — wrap with Firebase init
Future<void> main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await Firebase.initializeApp(options: DefaultFirebaseOptions.currentPlatform);

  // Catch all Flutter framework errors
  FlutterError.onError = FirebaseCrashlytics.instance.recordFlutterFatalError;

  // Catch all async errors outside Flutter framework
  PlatformDispatcher.instance.onError = (error, stack) {
    FirebaseCrashlytics.instance.recordError(error, stack, fatal: true);
    return true;
  };

  runApp(const ProviderScope(child: MyApp()));
}
```

#### Sentry for Flutter (Tier 2+ — unified with backend)
If using Sentry for backend, add Flutter SDK for unified error tracking across all layers:
```yaml
dependencies:
  sentry_flutter: ^8.x
```

```dart
await SentryFlutter.init(
  (options) {
    options.dsn = const String.fromEnvironment('SENTRY_DSN');
    options.environment = const String.fromEnvironment('ENV', defaultValue: 'production');
    options.tracesSampleRate = 0.1;  // 10% of transactions
  },
  appRunner: () => runApp(const ProviderScope(child: MyApp())),
);
```

Advantage: one Sentry project shows errors from backend + mobile with shared user context — when a user crashes, you can see the API calls that preceded it.

## Alerts: What to Alert On

Minimum viable alert set for any project:

| Alert | Threshold | Channel | Why |
|---|---|---|---|
| API 5xx rate | >10 errors/min | Email + Slack | Users hitting errors |
| API p99 latency | >2000ms | Email | Degraded UX |
| ECS CPU | >80% for 3 min | Email | Scaling needed |
| RDS connections | >80% of max | Email | Connection exhaustion |
| RDS storage | >80% used | Email | Disk full imminent |
| Sentry error spike | >2x baseline | Sentry | New bug shipped |
| Uptime | Any downtime >1min | SMS/PagerDuty | Total outage |

Anti-pattern: alerting on everything = alert fatigue = ignored alerts = missed real outages.

## Ansible — Does It Fit Here?

Short answer: **No, not for this factory's stack.** Here's why:

Ansible is a configuration management tool — it manages server state (install packages, configure SSH, set up users). This factory uses:
- **ECS Fargate** (serverless containers — no servers to configure)
- **RDS** (managed database — no server to SSH into)
- **S3 + CloudFront** (fully managed)

There's nothing to run Ansible against. OpenTofu already handles infrastructure provisioning, and Docker handles application configuration.

Ansible belongs in stacks that use EC2 instances or bare metal servers. If a project graduates to that (e.g., needs a dedicated worker server), add Ansible then.

## Summary Decision Table

```
Starting a new project       → Tier 1 (CloudWatch + Sentry free + Firebase Crashlytics)
First paying customers       → Upgrade to Tier 2 (add Grafana Cloud)
Scaling up / on-call         → Tier 3 (Datadog or Grafana Cloud Pro)
Need audit trail             → CloudTrail always (Tier 1+)
Mobile crashes               → Firebase Crashlytics always (free, best Flutter integration)
Unified errors API + mobile  → Sentry (same SDK, shared context)
Traces                       → OpenTelemetry (instrument once, choose backend later)
Server configuration         → Ansible NOT needed for this stack (no EC2/bare metal)
```
