# Hala Dashboard — Project Skill

A React 19 + Vite + TypeScript admin dashboard using TanStack Router, TanStack Query, React Hook Form, Zod, Tailwind CSS v4, and Radix UI / shadcn-style components.

---

## Tech Stack

| Concern        | Library                                      |
|----------------|----------------------------------------------|
| Bundler        | Vite 7                                       |
| Language       | TypeScript 5.8 (strict)                      |
| Routing        | TanStack Router (file-based)                 |
| Data fetching  | TanStack Query v5                            |
| Forms          | React Hook Form v7 + Zod v4                  |
| HTTP           | Axios (instance at `src/lib/axios.ts`)       |
| Styling        | Tailwind CSS v4 (utility classes, no modules)|
| UI primitives  | Radix UI + shadcn components in `src/components/ui/` |
| Icons          | Phosphor Icons + Tabler Icons + Lucide       |
| Tables         | TanStack Table v8                            |
| State          | Zustand (only when local state is not enough)|
| Linting/Format | Biome                                        |
| Toasts         | Sonner                                       |

## File Naming Conventions

- **All files and folders**: `kebab-case`
- **Private/local files** (not imported outside the folder): prefix with `-` (e.g. `-form.tsx`, `-schema.tsx`, `-service.ts`)
- **Public entry points**: `index.tsx` or `index.ts`
- Route params use PascalCase prefixed with `$` (e.g. `$BlogId`, `$EventId`)

## Routing

Routes use **TanStack Router file-based routing** under `src/routes/`.

```
src/routes/
├── __root.tsx
├── login/index.tsx
└── dashboard/
    ├── route.tsx                        # Dashboard layout
    ├── index.tsx                        # /dashboard
    └── <module>/
        ├── index.tsx                    # /dashboard/<module>
        ├── create/index.tsx
        ├── update/$ParamId/index.tsx
        └── $ParamId/index.tsx           # view
```

- Route files are **thin** — just import and render from `src/app/`
- Never edit `routeTree.gen.ts` — it is auto-generated by the Vite plugin
- Route params: `$PascalCaseId` (e.g. `$BlogId`, `$AdminId`)

**Route file example:**
```tsx
import { createFileRoute } from '@tanstack/react-router'
import BlogPage from '@/app/blog'

export const Route = createFileRoute('/dashboard/blog/')({
  component: BlogPage,
})
```

## Data Fetching (TanStack Query)

Query keys live in `src/lib/query-keys.ts` — always add new keys there.

**useQuery (list):**
```ts
import { useQuery } from '@tanstack/react-query'
import api from '@/lib/axios'
import { queryKeys } from '@/lib/query-keys'

export const useModuleList = (params?: Record<string, unknown>) =>
  useQuery({
    queryKey: queryKeys.module.list(params),
    queryFn: async () => {
      const { data } = await api.get('/module', { params })
      return data
    },
  })
```

**useMutation:**
```ts
import { useMutation, useQueryClient } from '@tanstack/react-query'
import api from '@/lib/axios'
import { queryKeys } from '@/lib/query-keys'

export const useCreateModule = () => {
  const queryClient = useQueryClient()
  return useMutation({
    mutationFn: async (payload: CreateModulePayload) => {
      const { data } = await api.post('/module', payload)
      return data
    },
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: queryKeys.module.all })
    },
  })
}
```

## Forms (React Hook Form + Zod)

**Schema:**
```tsx
// -schema.tsx
import { z } from 'zod'

export const createModuleSchema = z.object({
  name: z.string().min(1),
})
export type CreateModuleValues = z.infer<typeof createModuleSchema>
```

**Form component:**
```tsx
// -create-form.tsx
import { useForm } from 'react-hook-form'
import { zodResolver } from '@hookform/resolvers/zod'
import { Form, FormField, FormItem, FormLabel, FormControl, FormMessage } from '@/components/ui/form'
import { Input } from '@/components/ui/input'
import { Button } from '@/components/ui/button'
import { createModuleSchema, type CreateModuleValues } from './-schema'
import { useCreateModule } from './-service'

export function CreateModuleForm() {
  const { mutate, isPending } = useCreateModule()
  const form = useForm<CreateModuleValues>({
    resolver: zodResolver(createModuleSchema),
    defaultValues: { name: '' },
  })

  return (
    <Form {...form}>
      <form onSubmit={form.handleSubmit((v) => mutate(v))} className="space-y-4">
        <FormField control={form.control} name="name" render={({ field }) => (
          <FormItem>
            <FormLabel>Name</FormLabel>
            <FormControl><Input {...field} /></FormControl>
            <FormMessage />
          </FormItem>
        )} />
        <Button type="submit" disabled={isPending}>
          {isPending ? 'Saving...' : 'Create'}
        </Button>
      </form>
    </Form>
  )
}
```

## Null Safety in Tables (CRITICAL)

All property access in table cell renderers MUST use optional chaining. This prevents crashes when API returns null/undefined nested objects.

```typescript
// ❌ WRONG — crashes if resident is null
{ id: 'resident', header: 'Resident', cell: ({ row }) => row.original.resident.name }

// ✅ CORRECT — handles null safely
{ id: 'resident', header: 'Resident', cell: ({ row }) => row.original.resident?.name ?? '—' }

// ✅ CORRECT — with styled fallback
{ 
  id: 'resident', 
  header: 'Resident', 
  cell: ({ row }) => row.original.resident?.name ?? <span className="text-[var(--text-muted)]">—</span> 
}
```

**Apply to ALL of these:**
- Nested object access: `row.original.user?.name`
- Array access: `row.original.units?.length ?? 0`
- Date formatting: `row.original.createdAt ? formatDate(row.original.createdAt) : '—'`
- Enum lookups: `statusVariant[row.original.status ?? 'unknown']`

## Safe Date Formatting

The `formatDate()` utility must handle edge cases:

```typescript
// src/lib/utils.ts
export function formatDate(date: string | Date, opts?: Intl.DateTimeFormatOptions) {
  try {
    // dateStyle/timeStyle CANNOT be mixed with year/month/day
    const hasStyleShorthand = opts?.dateStyle || opts?.timeStyle;
    const finalOpts: Intl.DateTimeFormatOptions = hasStyleShorthand
      ? opts!
      : { year: 'numeric', month: 'short', day: 'numeric', ...opts };
    return new Intl.DateTimeFormat('en-US', finalOpts).format(new Date(date));
  } catch {
    return String(date).slice(0, 10);
  }
}
```

**Why this matters:**
- `formatDate(date, { dateStyle: 'medium', timeStyle: 'short' })` + default `{ year, month, day }` = **RangeError crash**
- The try/catch ensures malformed dates don't crash the entire page

## TypeScript Types Must Match API

Your TypeScript interfaces MUST exactly match what the backend API returns.

```typescript
// ❌ WRONG — frontend type doesn't match API
// API returns: { user: { name: 'John' } }
// Frontend expects:
interface Bill {
  resident: { name: string };  // Different field name!
}

// ✅ CORRECT — Either:
// Option A: Backend maps user → resident before returning
// Option B: Frontend type matches API exactly
interface Bill {
  user: { name: string };  // Matches API
}
```

**Common mismatches to watch for:**
- `user` vs `resident` vs `customer`
- `notes` vs `adminNotes` vs `adminNote`
- `assignments` vs `residents` (flattened arrays)
- Enum casing: `IN_PROGRESS` vs `in_progress`
