# Secrets Management — Generation and Handling

---

## Golden Rules

1. **Never generate secrets manually** — always use OpenTofu `random_password` or `random_bytes`
2. **Never store secrets in files that touch git** — `.env`, `.tfvars`, source code
3. **Never log secrets** — not even in debug mode
4. **Never return secrets in API responses** — including password hashes
5. **Never put secrets in ECS `environment` field** — use `secrets` field (Secrets Manager)
6. **Rotate after bootstrap** — personal tokens used for setup should be replaced after project is live

---

## Secret Inventory

Every project generates these secrets. All live in Secrets Manager at `/{project}/{env}/app`.

| Secret | Type | Length | Generated by |
|---|---|---|---|
| JWT_SECRET | random string | 64 chars | `random_password` |
| DB_PASSWORD | random string | 32 chars, no special | `random_password` |
| DATABASE_URL | connection string | built from above | `local` in OpenTofu |
| ENCRYPTION_KEY | random bytes | 32 bytes (256-bit) | `random_bytes` |

For features that need additional secrets:
| Feature | Secret | Type |
|---|---|---|
| Email | SMTP_PASSWORD | From email provider |
| Stripe | STRIPE_SECRET_KEY | From Stripe dashboard |
| FCM (push) | FCM_SERVER_KEY | From Firebase console |

## Secrets Access Hierarchy

```
OpenTofu (infra creation) — full access to create/update secrets
    │
    └── Writes to: AWS Secrets Manager

ECS Task Role — read-only access to specific secret ARN
    │
    └── Reads from: Secrets Manager at container start
    └── Injected as: environment variables in container

Application code — reads from process.env
    │
    └── JWT_SECRET, DATABASE_URL available as env vars
    └── Never reads from Secrets Manager directly (ECS does it)

GitHub Actions — access via GitHub Secrets
    │
    └── AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, CF_API_TOKEN
    └── Set via: gh secret set NAME
    └── Referenced as: ${{ secrets.NAME }}
```

## Local Development

Local `.env.local` file (gitignored):

```bash
# .env.local — gitignored, never committed
NODE_ENV=development
PORT=3000
JWT_SECRET=local-development-jwt-secret-minimum-64-chars-for-testing-only-1234
DATABASE_URL=postgresql://postgres:postgres@localhost:5432/myproject_dev
ALLOWED_ORIGINS=http://localhost:5173
```

`.gitignore` must include:
```gitignore
.env
.env.local
.env.*.local
*.tfvars
!*.tfvars.example
```

## Secret Rotation Procedure

After the project is live:

### 1. Rotate Bootstrap Credentials
These were used during setup and should be replaced with project-specific ones:

```bash
# GitHub: Create new PAT scoped only to this repo
# Update: gh secret set GITHUB_TOKEN --body "new-token"

# Cloudflare: Create new API token for this project only
# Update: gh secret set CF_API_TOKEN --body "new-token"

# AWS: Create new IAM user for CI/CD with minimal permissions
# Update: gh secret set AWS_ACCESS_KEY_ID and AWS_SECRET_ACCESS_KEY
```

### 2. Rotate Application Secrets
```bash
# Update secret in Secrets Manager
aws secretsmanager put-secret-value \
  --secret-id "/myproject/production/app" \
  --secret-string "$(aws secretsmanager get-secret-value \
    --secret-id /myproject/production/app \
    --query SecretString --output text | jq '. + {"JWT_SECRET":"new-value"}')"

# Force ECS to pick up new secret (redeploy)
aws ecs update-service \
  --cluster myproject-production \
  --service myproject-api \
  --force-new-deployment
```

## Common Secret Leaks — How to Detect

```bash
# Scan for potential secrets in git history
git log --all --full-history --diff-filter=A -p -- '*.env*' '*.key' '*.pem'

# Scan current files for patterns
grep -r "AKIA" .  # AWS Access Key IDs start with AKIA
grep -r "password.*=" --include="*.js" --include="*.py" --include="*.ts" .

# Use tools
npx @secretlint/secretlint "**/*"
```

If a secret is found committed to git:
1. Immediately rotate/revoke the exposed secret
2. Remove from git history: `git filter-branch` or `git filter-repo`
3. Force-push (requires bypass of branch protection)
4. Document in incident log
