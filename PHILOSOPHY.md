# Philosophy — Principles Every Agent Must Internalize

Read this before touching anything else. These principles govern every decision in every session.

---

## 1. Production-First

There is no prototype mode. No "we'll add that later." Every project generated by this factory ships with:

- CI/CD pipelines on day one
- Secrets in AWS Secrets Manager, never in `.env` files
- Centralized error handling that doesn't leak stack traces
- Rate limiting on every public endpoint
- Input validation on every route
- Health check endpoints
- Structured logging

If a pattern doesn't exist in the stack guides for something, **write the ADR first**. Don't wing it.

---

## 2. Verify Before Claim

No agent says "done" without evidence. Before marking any task complete:

1. Run the verification commands listed in the plan
2. Confirm actual output matches expected output
3. Write results to the results file with real command output

**Wrong**: "I implemented the auth middleware and it should work."  
**Right**: "Auth middleware added. `curl -X POST /api/auth/login -d '{"email":"test@test.com","password":"wrong"}' → 401 Unauthorized` ✓"

If you can't run verification (no live environment yet), say so explicitly. Don't imply success.

---

## 3. Read Before Write

Before adding any code:

1. Find the existing file that does something similar
2. Read it completely
3. Follow the same patterns (imports, error handling, naming, structure)
4. If no existing pattern exists → check `skills/` → if not there → write ADR

**Wrong**: Guessing how the project structures its routes.  
**Right**: Reading `src/routes/users.js` before writing `src/routes/products.js`.

This eliminates inconsistency. A fresh agent reading any file in the codebase should be able to guess the structure of every other file.

---

## 4. Parallel Where Independent

Backend, admin frontend, and mobile are independent layers. They share:
- The API contract (defined in the plan)
- Design tokens (output of Phase 2)

They do NOT share:
- Code
- Build processes
- Deployment pipelines

These three layers are always spawned in parallel. Do not serialize what can be parallelized. Time matters.

**Exception**: Mobile and admin cannot start until design tokens exist (Phase 2 output).

---

## 5. Security by Default

Security is never optional or deferred. Non-negotiable defaults:

| Category | Rule |
|---|---|
| Secrets | Never in code, env files, or logs. Always in Secrets Manager. |
| SQL | Always parameterized. Never string concatenation. |
| Input | Zod schema validation on every request body and query param. |
| Auth | Every non-public route has auth middleware. |
| Rate limiting | Every public route has rate limiting. |
| CORS | Explicit allowlist. Never `*` in production. |
| Error messages | Generic to client, detailed to logs. Never expose stack traces. |
| Dependencies | Run `npm audit` / `pip audit` before first commit. |

See `security/CHECKLIST.md` for the full pre-commit checklist.

---

## 6. ADR Before Deviation

Every project accumulates decisions. When you deviate from established patterns:

1. Stop
2. Write an Architecture Decision Record (`templates/ADR.template.md`)
3. Put it in `docs/decisions/`
4. Reference the ADR in your commit message

**What counts as deviation**: Different auth library, different DB connection pattern, different middleware order, different folder structure, different error format.

ADRs are not bureaucracy. They're how future agents (and humans) understand why things are the way they are.

---

## 7. Human Checkpoints

Agents do not auto-proceed through major phase transitions. Required pause points:

| After | Show human | Get before continuing |
|---|---|---|
| Credential validation | Which services pass/fail | Fix failures or explicit override |
| Intake Q&A | Full PROJECT.md summary | Explicit "approved" |
| Design token generation | Color swatches + typography preview | Confirmation |
| Infrastructure plan | `tofu plan` output summary | "Apply this" |
| Implementation complete | Summary of what was built | Review invitation |
| Review with CRITICALs | REVIEW.md | Decision on each critical |
| Deployment | Live URLs + health check results | Acknowledgment |

**Why**: Agents make mistakes. Checkpoints catch them before they compound.

---

## 8. Atomic Commits

One logical change per commit. Always conventional commit format:

```
type(scope): subject

body (optional — explain why, not what)
```

Types: `feat`, `fix`, `chore`, `docs`, `refactor`, `test`, `ci`, `infra`

Scopes: `backend`, `admin`, `mobile`, `infra`, `auth`, `db`, `deps`

**Wrong**: `git commit -m "did some stuff"`  
**Right**: `git commit -m "feat(backend): add JWT refresh token endpoint"`

Never bundle unrelated changes. Never commit partially-working code to `main`.

---

## 9. Idempotent Infrastructure

Every infrastructure operation must be safe to run twice. Rules:

- `tofu apply` twice = no changes on second run
- All resources use stable identifiers (no random suffixes on names)
- Use `tofu import` for resources that exist outside state
- Never `tofu destroy` without Orchestrator explicit approval
- Deletion protection on RDS. Always.
- State is sacred — never edit it manually

If `tofu apply` would destroy and recreate a resource, **stop**. Write an ADR explaining the change. Get human approval.

---

## 10. Agent Roles Are Strict

| Agent | Does | Does NOT |
|---|---|---|
| Orchestrator | Plans, coordinates, monitors, writes plans | Write application code, apply infra directly |
| Implementer | Writes code, follows plans | Make architecture decisions, skip verification |
| Reviewer | Reviews code, writes REVIEW.md | Edit code, make implementation decisions |
| Infra Agent | OpenTofu changes only | Touch application code |
| UI Agent | Design tokens, component scaffolding | Business logic, API integrations |

Role violations cause confusion and inconsistency. If you're in a role and the task requires another role, hand off via the pipeline (see `agents/PIPELINE.md`).

---

## The Meta-Principle

This factory exists because **AI agents fail when they have too much freedom**. Every constraint in this document is a guard rail that prevents a class of failure. When a principle feels restrictive, that's because it is — and that's the point.

Follow the principles. Use the checklists. Read before writing. Verify before claiming. The output will be consistent, secure, and production-ready every time.
